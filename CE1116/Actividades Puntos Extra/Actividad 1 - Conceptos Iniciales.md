# 📚 Mis Conceptos de Ingeniería de Software

## 1. Interesados y Requerimientos

> Conceptos fundamentales para entender qué se necesita construir y para quién.

| Término | Definición |
|---|---|
| **Stakeholder** | Entidad involucrada en un proyecto o sus resultados *(ej: cliente, desarrolladores, gerentes)* |
| **Requerimientos de Negocio** | Objetivos de alto nivel que una organización debe cumplir |
| **Requerimientos de Sistema** | Especificaciones de lo que el sistema/producto debe cumplir |
| **Requerimientos Funcionales** | Funciones que el sistema debe ser capaz de realizar |
| **Requerimientos No Funcionales** | Características de calidad del producto *(ej: seguridad, eficiencia, escalabilidad)* |

---

## 2. Scrum

> Marco de trabajo ágil basado en ciclos cortos e iterativos llamados *sprints*.

### Filosofía y Marco
- **Filosofía Ágil** — Metodología iterativa enfocada en aprendizaje constante y respuesta rápida al cambio.
- **Scrum** — Uno de los esquemas de metodología ágil, organizado en torno a *sprints*.

### Roles
- **Product Owner** — Relaciona al equipo con el cliente, traduce sus deseos en tareas concretas.

### Artefactos
- **Product Backlog** — Lista completa de todo lo que el sistema necesita hacer.
- **Sprint Backlog** — Subconjunto del backlog asignado a un sprint específico.
- **Taskboard** — Tablero con tareas organizadas por estado: `To Do` → `Doing` → `Done`

### Ceremonias
- **Sprint Planning** — Reunión de equipo para planificar y organizar el próximo sprint.

## 3. Estimación y Ejecución

> Herramientas y criterios para planificar el trabajo y asegurar su calidad.

| Término                      | Definición                                                                            |
| ---------------------------- | ------------------------------------------------------------------------------------- |
| **User Story**               | Descripción de una funcionalidad deseada desde el punto de vista del usuario          |
| **Story Points**             | Unidad de medida para estimar el esfuerzo requerido para completar una *User Story*   |
| **Planning Poker**           | Dinámica en la que el equipo asigna estimaciones de forma anónima y luego las discute |
| **Velocity**                 | Promedio de trabajo completado que permite predecir la capacidad futura del equipo    |
| **Definition of Done (DoD)** | Criterios que deben cumplirse para considerar una tarea como completada               |
| **Pruebas Funcionales**      | Verificaciones para asegurar que el software cumple los requerimientos definidos      |
